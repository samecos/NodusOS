# Nodus 概要设计文档 (High-Level Design)

> 版本: v1.0 | 状态: Approved | 日期: 2026-05-04

---

## 1. 引言

### 1.1 文档目的

本文档描述 Nodus（结绳）AI原生操作系统的概要架构设计，包括系统分层、模块分解、技术选型、关键设计决策及其理由。目标读者为架构师、技术负责人和核心开发者。

### 1.2 系统概述

Nodus 是一个面向开发者的 AI 原生操作系统。与传统 OS 不同，Nodus 的核心路由是自然语言意图理解引擎，而非文件系统和窗口管理器。MVP 版本聚焦于两个基石能力：代码库语义索引（脑）和环境全托管（手脚）。

### 1.3 设计目标

| 目标 | 指标 | 优先级 |
|------|------|--------|
| 语义理解完整性 | 符号索引覆盖率 ≥ 95%，调用图准确率 ≥ 95% | P0 |
| 查询响应速度 | 意图到结果卡片 ≤ 3s | P0 |
| 环境自动化 | 项目打开到就绪 ≤ 2min，零人工命令 | P0 |
| 本地运行 | MVP 不依赖任何云服务 | P0 |
| 架构可扩展性 | 新语言支持通过解析器插件接入，不修改核心 | P1 |
| 部署简化 | 单机桌面应用，零服务端依赖 | P1 |
| 原生依赖可移植性 | `better-sqlite3` / `tree-sitter` 在目标平台可正常加载 | P1 |

### 1.4 术语

| 术语 | 定义 |
|------|------|
| Module | 系统中的独立功能单元，通过 trait 暴露接口 |
| Event Bus | 模块间松散通信的发布/订阅机制 |
| Card | 按需生成、用完即焚的 UI 单元 |
| Skill/Capability | 系统可调用的后台能力组件 |
| Semantic Index | 代码库的符号、引用、调用图的总和 |

## 2. 架构风格

### 2.1 整体风格：模块化单体 (Modular Monolith) + 事件驱动

MVP 阶段采用**模块化单体**架构。所有模块在同一进程中运行，通过 Rust trait 接口直接调用，辅以事件总线做松散通信。此选择在 MVP 阶段兼顾开发速度和模块边界清晰度，同时保留未来拆分为独立微服务的可能。

```
┌────────────────────────────────────────────────┐
│                  Nodus Shell                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │
│  │ Intent   │ │ Context  │ │              │  │
│  │ Engine   │ │ Manager  │ │  Event Bus   │  │
│  └──────────┘ └──────────┘ │  (Pub/Sub)   │  │
│  ┌──────────┐ ┌──────────┐ │              │  │
│  │ Code     │ │ Env Mgr  │ │              │  │
│  │ Intel    │ │          │ │              │  │
│  └──────────┘ └──────────┘ │              │  │
│  ┌──────────┐ ┌──────────┐ │              │  │
│  │ Git      │ │ File     │ │              │  │
│  │ Intel    │ │ Watcher  │ │              │  │
│  └──────────┘ └──────────┘ └──────────────┘  │
│  ┌──────────┐ ┌──────────┐                    │
│  │ Voice    │ │ UI       │                    │
│  │ Pipeline │ │ Renderer │                    │
│  └──────────┘ └──────────┘                    │
│  ┌──────────────────────────┐                 │
│  │     Knowledge Store       │                 │
│  └──────────────────────────┘                 │
└────────────────────────────────────────────────┘
```

### 2.2 设计原则

1. **接口即契约** — 每个模块对外暴露 trait，内部实现可随意重构
2. **上层依赖下层，下层不感知上层** — 严格分层，避免循环依赖
3. **同步查询，异步索引** — 用户查询走同步快路径；索引构建异步后台执行
4. **语言无关核心 + 插件式解析器** — 核心索引引擎不依赖特定编程语言

## 4. 模块分解

| 模块 | 层级 | 职责 | 核心依赖 |
|------|------|------|---------|
| **Nodus Shell** | 框架 | 应用生命周期、模块注册、事件总线 | 所有模块 |
| **Intent Engine** | 编排 | NLU 意图解析：分类 + 实体提取 + 歧义判断 | Context Manager |
| **Context Manager** | 编排 | 追踪开发者的文件、光标、选中、查询历史 | 无 |
| **Code Intelligence** | 能力 | 语义索引：解析、符号提取、引用解析、调用图 | Knowledge Store, Language Parsers |
| **Environment Manager** | 能力 | 运行时检测、安装、依赖管理 | 无 |
| **Git Intelligence** | 能力 | Git 操作封装：log, diff, blame | 无 |
| **File Watcher** | 能力 | 文件系统事件监听，触发增量索引 | 无 |
| **UI Renderer** | 界面 | 卡片渲染、代码查看、呼吸灯、输入条 | 所有能力模块接口 |
| **Voice Pipeline** | 界面 | 唤醒词检测、STT、TTS、无声模式 | 无 |
| **Knowledge Store** | 数据 | SQLite + 内存索引的持久化层 | 无 |

## 5. 技术选型

### 5.1 应用外壳

| 选项 | 决策 | 理由 |
|------|------|------|
| Tauri (Rust + Web前端) | **推荐** | 与长期Rust内核方向一致；包体小（<10MB）；内存安全 |
| Electron | 备选 | 生态成熟、开发快，但包体大（>150MB） |
| 纯Rust GUI | 不选 | UI开发效率不足以支撑复杂的卡片渲染 |

**决策**：Tauri。后端Rust承担Code Intelligence等计算密集模块，前端React渲染卡片和代码。

### 5.2 代码解析

| 选项 | 决策 | 理由 |
|------|------|------|
| tree-sitter | **唯一选择** | 语言无关、增量解析、Rust绑定一流、社区语法文件覆盖JS/TS/Python |

### 5.3 存储

| 组件 | 方案 | 理由 |
|------|------|------|
| 结构化索引数据 | SQLite | 嵌入式、零配置、查询优化成熟 |
| 查询热路径 | 内存索引 + SQLite持久化 | 毫秒级查询，SQLite做持久化和故障恢复 |

### 5.4 语音

| 组件 | 方案 | 理由 |
|------|------|------|
| 唤醒词 | 系统原生API (Apple Speech / Windows Speech) | MVP快速验证；延迟低 |
| STT | 同上 | 无需部署模型 |
| 长期方向 | Whisper.cpp (本地) | 隐私、离线、跨平台一致 |

### 5.5 UI

| 组件 | 方案 |
|------|------|
| 卡片/UI框架 | React 18 + SVG (调用图) + CSS (卡片) |
| 代码编辑器 | Monaco Editor (VSCode内核) |
| 构建工具 | Vite |

## 7. 架构决策记录 (ADR)

### ADR-001: 模块化单体而非微服务

**状态**：已接受

**上下文**：MVP 阶段团队规模小，所有功能在单机上运行。

**决策**：采用模块化单体架构。模块间通过 Rust trait 通信，辅以事件总线。

**后果**：
- 优点：开发速度快，调试简单，无网络通信开销
- 风险：模块边界可能逐渐模糊，需要代码审查维护
- 缓解：严格文件夹边界，trait 作为模块间的唯一接触点

### ADR-002: 本地优先，零云依赖

**状态**：已接受

**上下文**：开发者对代码隐私高度敏感。MVP 需要极低的试用门槛。

**决策**：所有计算在本地完成。语义索引在本地构建，语音识别使用系统原生 API 或本地模型，不向任何云端发送源代码。

**后果**：
- 优点：隐私保护、离线可用、零网络延迟
- 风险：本地模型精度可能不如云端大模型（Whisper.cpp vs 云端 STT）
- 缓解：使用系统原生 STT API（Apple/Windows 的云端转写质量很高）

### ADR-003: 意图解析同步，索引异步

**状态**：已接受

**上下文**：用户查询时需要即时响应。索引构建可以后台完成。

**决策**：IntentEngine::parse 为同步方法（<200ms，本地小模型）。CodeIntelligence::index_project 和 index_file 为异步方法，在 tokio runtime 中执行。

**后果**：
- 优点：用户查询路径无 async 开销，响应可预测
- 风险：parse 超过 200ms 会阻塞事件循环
- 缓解：意图分类使用轻量级本地模型（如 BERT-tiny），严格性能测试

### ADR-004: SQLite + 内存索引双层存储

**状态**：已接受

**上下文**：查询需要毫秒级响应，但数据需要在应用重启后保持。

**决策**：查询热路径走内存中的 HashMap/BTreeMap 索引。SQLite 做持久化存储和恢复。应用重启时从 SQLite 重建内存索引。

**后果**：
- 优点：查询速度快（<1ms），重启可恢复
- 风险：内存占用随项目规模线性增长
- 缓解：10万行代码项目估计 20-50MB 内存索引，可接受

## 8. 横切关注点

### 8.1 日志策略

| 层级 | 日志框架 | 存储 |
|------|---------|------|
| Rust 后端 | `tracing` crate | ~/.nodus/logs/nodus.log |
| Web 前端 | console API → Tauri invoke | 合并到同一日志文件 |
| 级别 | DEBUG (开发) / INFO (发布) | 按日轮转，保留7天 |

### 8.2 安全模型

MVP 阶段的安全边界：

- **文件系统访问**：仅读写用户显式打开的项目目录和 ~/.nodus/
- **网络访问**：仅用于依赖安装（npm/pip），且通过用户已配置的代理
- **进程权限**：以当前用户权限运行，不要求 root/admin
- **数据存储**：SQLite 文件权限 0600，仅当前用户可读写

### 8.3 国际化

MVP 仅支持中文（zh-CN）和英文（en-US）。IntentEngine 的意图分类模型需要两种语言的训练数据。UI 文本通过 i18n key 管理，前端使用 react-i18next。

### 8.4 可观测性

| 维度 | 方案 |
|------|------|
| 性能追踪 | tracing spans 覆盖关键路径（索引、查询、STT） |
| 错误追踪 | 所有模块错误记录到日志 + 查询历史表 |
| 使用统计 | 查询历史表记录每次查询的意图类型、延迟、结果数 |
| 崩溃报告 | 本地 crash dump，不上传（隐私优先） |

## 9. 技术债务与风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| tree-sitter 解析精度不足 | 符号索引覆盖率不达标 | 中 | MVP前对10个开源项目做基准测试 |
| 系统STT API 跨平台不一致 | 用户体验差异大 | 高 | 早期切换到 Whisper.cpp |
| Monaco Editor 与 Tauri 集成复杂 | 代码查看器体验差 | 中 | 备选 CodeMirror 6 |
| 内存索引过大 | 大项目（>50万行）内存压力 | 低（MVP阶段） | 实现LRU淘汰策略；大项目分片索引 |
| 意图分类精度不足 | 歧义反问过于频繁 | 中 | MVP前用真实开发者查询做精度测试 |

> 更完整的后续需求、技术债务与版本路线图，见 [05-Future-Roadmap.md](05-Future-Roadmap.md)。
