# Nodus 用户验收标准 (User Acceptance Criteria)

> 版本: v1.0 | 日期: 2026-05-04

---

## 1. 验收总则

### 1.1 验收角色

| 角色 | 谁 | 验收什么 |
|------|-----|---------|
| **作者-开发者** | Nodus 的创建者本人 | 自己日常开发中使用 Nodus，持续一周无阻塞性问题 |
| **早期试用者** | 2-3 位信任的开发者朋友 | 安装 Nodus，完成 5 个指定任务，给出体验反馈 |
| **自动化验收** | CI/CD Pipeline | 全部 126 个测试用例通过；性能指标不退化 |

### 1.2 验收哲学

Nodus MVP 的验收标准不是"功能完备"，而是：

> **"作者自己愿意日常使用它，并且当别人问'这东西能用吗'时，你敢于说'来，我演示给你看'。"**

验收分两级：
- **MUST** — 没有这个不能叫 MVP。不满足则不可发布。
- **SHOULD** — 有已知限制但可接受。不满足需记录为已知问题。

## 2. 按用户故事的验收标准

### US-1: 早晨恢复 — "我昨天写到哪了"

**用户故事**: 打开 Nodus 后，系统立即告诉我昨天写到哪里了，无需手动翻 git log 和终端历史。

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-1.1 | 打开项目后 5 秒内，代码工作区显示上次关闭时的文件和光标位置 | MUST | 秒表计时 |
| AC-1.2 | 显示上次未提交的更改摘要（文件列表 + 变更行数） | MUST | 与 `git status` 对比 |
| AC-1.3 | 显示上次运行的测试结果（如果有） | SHOULD | 检查测试卡片 |
| AC-1.4 | 如果上次会话异常退出，提示用户并从最近的自动保存恢复 | SHOULD | 模拟崩溃测试 |

**用户说"验收通过"的条件**: 早上打开电脑，不用手动做任何事，10秒内想起昨天的工作状态。

---

### US-2: 代码库导航 — "这个函数被谁调用了"

**用户故事**: 用自然语言（语音或文字）查询代码库，获得结构化答案，不再 grep + 人肉拼图。

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-2.1 | 说"X在哪里定义的"，3秒内返回符号的定义位置（文件+行号） | MUST | 秒表，10次取 P95 |
| AC-2.2 | 说"X被哪些地方调用了"，返回完整引用列表（文件+行号+代码片段） | MUST | 人工对比 IDE "Find All References" |
| AC-2.3 | 说"X的调用链路"，返回树状调用图，可展开/折叠/点击跳转 | MUST | 人工验证调用图正确性 |
| AC-2.4 | 说"X文件里有哪些函数"，返回该文件所有符号的概览 | MUST | 人工验证 |
| AC-2.5 | 说"改了X会影响哪些文件"，返回影响范围和风险评级 | SHOULD | 人工验证 |
| AC-2.6 | 语音和文字查询结果完全一致 | MUST | 同一查询用两种输入方式对比 |
| AC-2.7 | 查询结果准确率 ≥ 95%（10次查询中最多1次结果不准确） | MUST | 人工评判 |

**用户说"验收通过"的条件**: 用 Nodus 做代码库导航比 grep 快 10 倍且结果更好，连续 3 天不再手动 grep 代码。

---

### US-3: 环境零配置 — "不用装环境了"

**用户故事**: 克隆一个项目后，系统自动安装运行环境和依赖，不需要输入任何命令。

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-3.1 | 打开 JS/TS 项目，自动检测并安装正确版本的 Node.js | MUST | 对比 package.json engines 字段 |
| AC-3.2 | 打开 Python 项目，自动检测并安装正确版本的 Python | MUST | 对比 pyproject.toml requires-python |
| AC-3.3 | 自动安装项目依赖（npm/pip），无需用户输入命令 | MUST | 全程录屏 |
| AC-3.4 | 标准项目（<500依赖）从打开到就绪 ≤ 2 分钟 | MUST | 秒表计时 |
| AC-3.5 | 环境状态可视化：显示进度和当前操作 | MUST | 观察环境卡片 |
| AC-3.6 | 安装失败时显示人类可理解的错误信息 + 手动修复建议 | MUST | 模拟网络中断 |
| AC-3.7 | 检测并适配已安装的运行时（不重复安装） | SHOULD | 检查重复安装 |
| AC-3.8 | 支持 npm/yarn/pnpm 和 pip/poetry/uv 的自动检测 | MUST | 6种包管理器各测1个项目 |

**用户说"验收通过"的条件**: 拿到一台新电脑，安装 Nodus 后，打开 3 个不同类型的项目，不用打开终端输入任何 `npm install` 或 `pip install`。

---

### US-4: 无声模式 — "会议室里也能用"

**用户故事**: 在开放办公室或会议室，用键盘输入意图，系统以卡片回应，不发出声音。

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-4.1 | Ctrl+Space 唤起意图输入条，输入文字回车发送 | MUST | 操作录屏 |
| AC-4.2 | 无声模式下唤醒词不响应 | MUST | 大声说唤醒词，确认不触发 |
| AC-4.3 | 无声模式和有声模式可手动切换 | MUST | 长按/语音命令切换各3次 |
| AC-4.4 | 切换模式时给出短暂视觉反馈 + 呼吸灯变化 | SHOULD | 观察提示条和呼吸灯 |
| AC-4.5 | 文字查询和语音查询的结果质量一致 | MUST | 同一查询对比 |

**用户说"验收通过"的条件**: 在会议室里打开 Nodus，全程不发出声音，完成一次代码查询。

---

### US-5: 模糊意图反问 — "不确定就问我"

**用户故事**: 当系统不确定我的意思时，生成选项让我选，而不是瞎猜。

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-5.1 | 信息不足时（模糊输入 + 无上下文）展示 2-3 个候选选项 | MUST | 10次模糊输入测试 |
| AC-5.2 | 每个选项描述清晰，能让用户判断区别 | MUST | 人工可读性评判 |
| AC-5.3 | 用户选择后立即执行，选项卡片消散，结果卡片浮现 | MUST | 操作录屏 |
| AC-5.4 | 有上下文时不过度反问（光标在符号上 + "这个被哪里调用了" 应直接解析） | MUST | 10次带上下文测试 |
| AC-5.5 | 完全不理解时说"抱歉，没听懂"，建议重新表述 | MUST | 输入无意义文本 |

**用户说"验收通过"的条件**: 模糊输入时系统反问而不是乱猜；清晰输入时系统直接执行而不是追问。

## 3. 非功能验收标准

### 3.1 性能

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-P1 | 意图解析延迟 ≤ 200ms (P95) | MUST | 100次查询的延迟分布 |
| AC-P2 | 查询响应延迟 ≤ 3s (从语音结束到卡片显示) | MUST | 10次查询取 P95 |
| AC-P3 | 10万行项目首次索引 ≤ 30s | MUST | 生成的项目测试 |
| AC-P4 | 项目打开到环境就绪 ≤ 2min | MUST | 3种项目类型各3次取平均 |
| AC-P5 | 内存占用 ≤ 200MB (10万行项目索引后) | SHOULD | 进程 RSS 测量 |
| AC-P6 | 应用冷启动 ≤ 3s | SHOULD | macOS 活动监视器 |

### 3.2 可靠性

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-R1 | 单文件语法错误不影响其他文件的索引和查询 | MUST | 人工测试 |
| AC-R2 | 数据库损坏后能自动重建索引（代码不受影响） | SHOULD | 删除 nodus.db 测试 |
| AC-R3 | 应用连续运行 8 小时不崩溃、不内存泄漏 | MUST | 长期运行监控 |
| AC-R4 | 索引过程中可正常查询（只是部分结果不可用） | SHOULD | 索引中途查询测试 |

### 3.3 可用性

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-U1 | 新用户无需阅读文档，通过自然语言完成首次查询 | MUST | 找一位未参与开发的朋友试用 |
| AC-U2 | 所有语音操作在无声模式下有等价的文字操作 | MUST | 功能对等检查表 |
| AC-U3 | 错误信息使用自然语言（非技术栈信息），普通开发者能理解 | MUST | 人工可读性评判 |
| AC-U4 | 应用 UI 不出现任何计算机内部概念（文件路径、端口号、进程名） | SHOULD | 逐屏检查 |

### 3.4 隐私与安全

| ID | 验收标准 | 级别 | 测量方法 |
|----|---------|------|---------|
| AC-S1 | 源代码不离开本地设备 | MUST | 网络流量监控（断网测试） |
| AC-S2 | 语音可配置使用本地模型（Whisper.cpp）而非云端 API | SHOULD | 配置检查 |
| AC-S3 | 数据库文件权限为 0600 | MUST | `ls -la ~/.nodus/nodus.db` |
| AC-S4 | 查询历史存储在本地，用户可清除 | MUST | 功能验证 |

## 4. 按模块的验收清单

### 4.1 Knowledge Store

- [ ] `symbols_upsert` 幂等：重复插入不产生重复行
- [ ] `symbols_remove` 级联：删除文件时引用同步清理
- [ ] `symbols_find_by_name` 支持 LIKE 模糊匹配
- [ ] `callgraph_get` 返回的节点和边与索引一致
- [ ] 内存索引在重启后从 SQLite 正确恢复

### 4.2 Code Intelligence

- [ ] TypeScript: 函数/类/接口/类型/变量的提取和引用
- [ ] JavaScript: 函数/类/变量的提取和引用
- [ ] Python: 函数/类/方法的提取和引用
- [ ] 调用图包含 caller 和 callee 双向
- [ ] 语法错误文件被跳过，不影响整体
- [ ] node_modules/ 等目录自动排除

### 4.3 Intent Engine

- [ ] 6 种意图类型全部能正确分类（中英文）
- [ ] 有上下文时自动补全隐式参数
- [ ] 无上下文且信息不足时反问
- [ ] 不支持的能力告知用户（而非假装理解）

### 4.4 Environment Manager

- [ ] 自动检测 npm/yarn/pnpm/pip/poetry/uv
- [ ] 运行时安装和依赖安装异步执行
- [ ] 安装进度实时可视化
- [ ] 失败时提供人类可读的原因和建议

### 4.5 Git Intelligence

- [ ] git log 支持文件/目录/时间/作者过滤
- [ ] git diff 正确展示文件级和行级变更
- [ ] git blame 返回正确的 commit 信息
- [ ] 非 git 仓库优雅降级

### 4.6 Voice Pipeline

- [ ] 唤醒词检测可靠（误触发率低）
- [ ] 唤醒后录音自动结束（VAD）
- [ ] STT 转写结果准确
- [ ] 无声模式正确切换

### 4.7 UI Renderer

- [ ] 调用图卡片：树状结构，可展开/折叠/跳转
- [ ] 引用列表卡片：按文件分组，标注问题
- [ ] 变更历史卡片：时间线展示
- [ ] 歧义卡片：用户可点击选择
- [ ] 环境卡片：进度条和状态展示
- [ ] 所有卡片 5 分钟无交互自动消散
- [ ] 呼吸灯 5 种状态视觉可区分

### 4.8 Event Bus / Shell

- [ ] 标准事件路由全部正确连接
- [ ] 模块生命周期管理正确（初始化顺序 + 优雅关闭）

## 5. MVP 发布决策矩阵

### 足以发布的信号（全部 MUST 满足）

- [ ] 作者本人持续使用 Nodus 进行日常开发 ≥ 7 天
- [ ] 期间未发生"必须回到 VSCode 终端手动 grep"的情况
- [ ] CI 全量测试绿灯（126 个用例全部通过）
- [ ] 性能指标全部达标（见 3.1 节）
- [ ] 至少 1 位早期试用者完成 5 个指定任务并评分 ≥ 4/5

### 阻塞发布的问题（任一出现则延期）

- [ ] 任何 MUST 级别的验收标准不满足
- [ ] 存在导致数据丢失或代码损坏的 bug
- [ ] 作者自己日常使用中遇到阻塞性问题
- [ ] 早期试用者中 ≥ 50% 无法独立完成核心任务

### 已知限制（可接受的发布时间点）

- [ ] 仅支持 JS/TS/Python 三种语言
- [ ] 不支持 AI 代码生成（v2 功能）
- [ ] 不支持跨设备同步
- [ ] 不支持团队协作
- [ ] 仅 macOS 平台测试过
- [ ] 语音识别依赖系统原生 API（非离线）

## 6. 验收仪式 (Acceptance Ritual)

MVP 验收以以下仪式进行：

```
1. 在干净的 macOS 设备上安装 Nodus
2. 打开一个此前从未在 Nodus 中打开过的 TypeScript 中型项目
3. 在不输入任何终端命令的情况下，完成环境配置
4. 用语音完成以下 5 个任务：
   a. 找到某个函数的定义位置
   b. 查看该函数的所有调用方
   c. 展示该函数的完整调用链路
   d. 查看某个模块最近一周的变更历史
   e. 分析修改某个接口的影响范围
5. 切换到无声模式，用文字完成任务 a 和 b
6. 观察所有卡片在 5 分钟后自动消散
7. 关闭 Nodus，重新打开，确认状态恢复
```

**全部 7 步通过 = MVP 验收通过。**
