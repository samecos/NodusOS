# 核心用户旅程

以下旅程全部以开发者（C类用户）为对象。每个旅程都展示Nodus核心原则的具体落地。

**两块基石**（MVP不可或缺）：
- **旅程二（代码库语义索引）**——系统的"脑"，没有它系统看不懂代码
- **旅程五（环境全托管）**——系统的"手脚"，没有它系统动不了代码。同时它决定了Nodus的试用门槛：开发者打开就能工作，不需要经历"装环境→配环境→修环境→放弃"的漏斗

---

## 旅程一：早晨恢复——"我昨天写到哪了"

### 传统OS（每日痛苦）

8:55 到工位，唤醒电脑。

面前：昨天的VSCode还开着，但你不知道自己在修什么bug了。3个终端标签：一个在跑dev server，一个在跑测试（上次跑了一半），一个在git bisect中间。浏览器里14个标签：GitHub PR、Jira ticket、StackOverflow、文档x3、还有两个你忘了为什么打开的。Slack上37条未读消息。你开始人肉拼凑——翻git log → 看最近改的文件 → 读Slack消息找上下文 → 回忆昨天下午到底在想什么。

9:12。你大概想起来你是在修auth模块的一个bug。你开始继续。但你知道你漏了什么——昨天下午那半小时的思考，没有一个字被记录下来。

### Nodus

你坐下来。屏幕从休眠中亮起。

"早。昨天的工作：你在修auth模块的token刷新bug。下班前跑了测试套件，3个case失败，都在refreshToken逻辑附近。夜里我把失败的日志和相关代码关联了一下，错误集中在token过期时间戳的边界处理上。要继续吗？"

屏幕上没有打开的窗口，没有几十个标签。只有昨天最后编辑的那个函数，光标还在你停下的位置。

你说："继续。"

终端恢复到昨天的状态。测试结果显示在旁边——3个红叉，但每个旁边都标注了Nodus的关联分析。git diff显示你昨天的未提交更改。

你进入心流。9:01。

**这个旅程体现的原则**：认知卸载（上下文重建由系统完成）、全局上下文（系统记录了代码+终端+git+时间的完整状态）、主动计算（系统夜里跑了测试并做了关联分析，但只是准备好，没有替你做决定）

## 旅程二：代码库导航——MVP第一场景

### 传统OS

"这个订单退款逻辑，到底被哪些地方调用了？"

你打开终端：`grep -r "refundOrder" src/` → 输出127行。你开始逐行看。有些是定义，有些是调用，有些是注释里的引用。你把看起来相关的文件逐个打开，试图在脑子里画调用图。你又搜了`grep -r "refund" src/ | grep -v test | grep -v node_modules` → 360行。你开始怀疑自己是不是应该装一个代码分析工具。你打开浏览器搜索"typescript call graph tool"。

35分钟后，你画出了调用图。但不完全确定有没有遗漏。

### Nodus

"'refundOrder'的完整调用链路是什么样的？有没有传入undefined的风险？"

系统（已经建立了整个代码库的语义索引）：

"refundOrder在14处被调用。核心调用链：

```
order.controller.ts:postRefund()
  → order.service.ts:processRefund()
    → payment.service.ts:refundOrder()
      → payment-gateway.ts:submitRefund()
```

14处调用中，有3处传入的amount参数可能为undefined：
- `bulk-refund.ts:67` — 从CSV解析的amount字段，CSV列可能缺失
- `webhook-stripe.ts:132` — Stripe事件体中的amount不是必填字段
- `admin-api.ts:89` — 查询参数解析，未做默认值处理

要逐一展开这3处吗？"

屏幕上没有grep的原始输出。而是结构化的调用图、风险标注、和可直接导航的入口。

你说："展开bulk-refund.ts。"

15秒。不是35分钟。

### 为什么这是MVP第一场景

这是从0到1的**认知能力基础**。如果Nodus不能语义理解代码库，后续所有场景（AI辅助重构、自动调试、跨文件修改）都建立不起来。代码库语义索引是开发者版Nodus的**内核**。

**这个旅程体现的原则**：认知卸载（脑内构图→系统呈现）、全局上下文（系统理解整个代码库的符号关系）、语音优先（用语音做导航查询，键盘只用于写代码和确认）

## 旅程三：跨域调试——"登录为什么有时500"

### 传统OS

11:23。Slack上运营消息："用户反馈登录有时候会失败，看后台有500。"

你的接下来2小时：

1. 打开日志平台（浏览器标签+1）→ 搜索错误日志 → 找到一些500错误
2. 打开监控平台（浏览器标签+1）→ 看错误率曲线 → 确认不是突发
3. 回到终端 → `git log --since="3 days ago" -- src/auth/` → 找最近的改动
4. 发现一个token库版本更新的commit
5. 打开token库的GitHub release notes（浏览器标签+1）→ 看changelog
6. 打开代码对比token新旧API → 发现参数签名变了
7. 回到编辑器改代码
8. 写测试
9. 跑到一半想起要重启服务

中间被打断3次（Slack消息），每次都要重新找回上下文。

### Nodus

11:23。你说："线上登录接口最近开始间歇500，查一下原因。"

系统：

"最近24小时`/auth/login`端点出现23次500错误，首次出现在昨天14:52。同一时刻，有一个token库从v3.2.1升级到v3.3.0的依赖变更合并到了主分支。

错误堆栈全部指向token.verify()方法——v3.3.0的这个方法的参数签名变了，从`verify(token, secret)`变成了`verify(token, { secret })`。登录代码里有两处调用还在用旧的单参数签名。

这是变更对比。（屏幕上展示v3.2.1和v3.3.0的verify方法签名diff，以及代码中两处需要修改的位置）

要我修复吗？"

30秒。不是2小时。

你确认了方案。系统修改了代码，跑完了测试——全绿。你确认提交。

**是什么让这成为可能**：Nodus同时看到了日志（错误500的时间线和堆栈）、代码变更（依赖更新的commit时间线）、代码库（verify的调用位置）。这三个信息域在传统OS中分别在三个不同的工具里，没有一个东西把它们连起来。人类脑子当胶水。Nodus把胶水变成了系统能力。

**这个旅程体现的原则**：认知卸载（跨域关联由系统完成）、全局上下文（日志+代码+git+依赖的联合理解）、人授权限（系统给出了分析但让你决定是否修复）、务实降级（如果根因不确定，系统会说"我怀疑是这里，但需要你确认"）

## 旅程四：协同开发——人类描述意图，AI执行，人类修正

### 传统OS

"需要给用户模型加一个软删除功能。"

1. 打开user.ts模型文件
2. 写deletedAt字段
3. 改所有查询加了where: { deletedAt: null }
4. 改delete方法从真删除变成update deletedAt
5. 想起来了——还要加restore方法
6. 写迁移文件
7. 写测试
8. 发现漏了一个查询没加过滤
9. 又漏了一个关联查询
10. 改完。30-40分钟。不确定有没有漏。

### Nodus

你说："给用户模型加软删除。所有查询默认过滤已删除记录。已有的删除操作改成软删。"

系统扫描代码库：

"需要修改的：
- user.ts模型：加deletedAt字段
- 迁移文件：新建
- 7个查询文件中涉及user表的地方：加过滤条件
- 3处真删除的调用：改成update deletedAt
- 加restore方法

另外，order表和audit_log表有user的外键关联——软删除后这两个表的行为需要确认。order表的策略是什么？级联？阻止删除？"

你："订单保留。删除时如果有未完成订单，阻止删除。"

系统生成修改。屏幕上不是"粘贴的代码块"——是结构化的diff，逐文件展示。

你逐文件审查。在`user.service.ts`的delete方法上，你手动改了——加了一个日志记录：

```typescript
// 你手动加的
await auditLog.record('user_soft_deleted', { userId, deletedBy: ctx.user.id });
```

**系统自动捕获了这个修改。**它生成标注：人类在AI生成的delete方法上补充了auditLog记录。这个模式被记录到你的偏好中——下次涉及数据修改的操作，系统会主动建议加审计日志。

**这个旅程体现的原则**：
- 双通道模型：人类描述意图 + AI执行铺路 + 人类审查修正
- 自动标注飞轮：人类的修正被捕获为训练信号
- 认知卸载：人类不需要grep所有查询文件、不需要记住所有调用方
- 人授权限：生成的修改以diff形式逐个呈现，人类逐项确认或修正
- 系统还主动提醒了关联影响（外键关联的表）

## 旅程五：环境零配置——没有"搭建开发环境"这件事

### 传统OS

新MacBook到了。你开始配开发环境：

1. 安装Homebrew（先去官网复制命令）
2. `brew install node` → 版本不对，项目用的是18
3. 装nvm → 装Node 18
4. `brew install python@3.12`
5. 装pipenv → 不对，项目用的是poetry → 装poetry
6. 配数据库 → `brew install postgresql@15` → 启动 → 创建数据库 → 配用户
7. 配redis → `brew install redis`
8. 克隆项目 → `npm install` → 报错，node-gyp缺Python → 配Python路径
9. 配.env文件（找同事要）
10. 跑起来了 → 但测试跑不通 → 数据库迁移没跑
11. 配CI/CD本地模拟 → 装act
12. 配Git代理（公司在国内）

2-4小时。如果一切顺利的话。
