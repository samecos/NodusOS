# 交互范式

## 总纲：不是聊天，不是桌面，是第三种东西

Nodus的交互既不是ChatGPT式的对话流，也不是Windows式的窗口桌面。它是第三种范式：

**"指挥官与参谋部"模型**——你说一句，系统理解、规划、执行，然后把结果推给你。不需要持续对话，不需要管理窗口。节奏是：**指令 → 执行 → 交付 → 确认**。循环结束。

---

## 一、入口：如何唤起Nodus

Nodus始终在——但不是始终在**说话**。它像空气，默认安静。唤起方式四种，适应不同场景：

### 1. 语音唤醒（主力入口）
- 唤醒词："Nodus" 或 "结绳"
- 唤醒后直接说意图，无需停顿："Nodus，refundOrder的调用链路是什么"
- 私密场景（独自办公）：直接外放对话
- 公共场景（会议室/开放工位）：自动进入**无声模式**——用文字/卡片交互，Nodus也以文字回应

### 2. 全局快捷键（键盘入口）
- 单键或组合键唤起（用户可自定义，默认建议 `Ctrl+Space`）
- 唤起后出现一个**单行意图输入条**（类似Spotlight，但接受自然语言而非命令语法）
- 输入文字意图，回车发送
- 适用于：不方便说话的场景、深夜、开放办公室

### 3. 触控手势（移动/平板场景）
- 双指从屏幕底部上滑 → 唤起意图输入
- 单指长按屏幕任意位置 → 触发"这是什么"上下文感知

### 4. 上下文自动唤起（主动计算触发）
- 系统在已授权领域检测到需要用户注意的事件时，以最低打扰级别呈现
- 不是弹窗——是屏幕边缘的一个**微弱呼吸灯**或状态栏的**微小气泡**
- 用户决定何时处理

### 入口设计原则
- **语音是第一公民**，但不是唯一通道
- **无声模式是一等公民**——不只是"静音"，而是完整的文字交互模式
- 入口本身不应该占屏幕空间——唤起时才出现，用完消失

## 二、意图表达：人类如何告诉Nodus要什么

### 意图类别与最佳表达方式

| 意图类型 | 示例 | 最佳表达方式 | 原因 |
|---------|------|-------------|------|
| 导航型 | "跳转到auth模块" | 语音 / 文字 | 快于鼠标点击 |
| 查询型 | "refundOrder的调用链路" | 语音 / 文字 | 自然语言比grep语法快 |
| 编排型 | "跑测试，通过了就打staging" | 语音 / 文字 | 组合指令，不想要脚本 |
| 创作型 | 写代码、改代码 | 键盘 | 代码不适合用嘴说 |
| 审查型 | 看diff、确认修改 | 界面（卡片）+ 键盘 | 需要仔细看，需要编辑 |

### 复合意图：语音 + 选中 + 语音

最高效的开发者交互不是"纯语音"或"纯键盘"，而是复合：

> 用鼠标/键盘选中一段代码 → "Nodus，给这段代码加上空值检查" → Nodus在选中的代码处生成修改

> 光标停在某个函数上 → "Nodus，这个函数在哪些地方被调用" → Nodus展示调用图

**上下文感知的原则**：如果用户当前有选中的代码、光标所在位置、或正在看的内容，这些自动成为意图的**隐式参数**。不需要每次都说"这个函数""那段代码"。

### 模糊意图的处理

当Nodus不确定用户意思时，**不猜测，不假装理解，而是反询问**：

用户："把那边的那个改一下。"

Nodus（屏幕上浮出3个卡片选项）：
- A：把当前选中的函数参数类型改成interface
- B：把上次修改的文件中的TODO注释改掉
- C：把刚才测试失败的那个断言改一下

用户："A。"（或点击卡片A，或说"A"）

这个反询问本身就是意图澄清过程。而且用户的这次选择（"模糊时用户通常指当前上下文"）被系统学习——下一次，类似场景下Nodus会更倾向A类解释。

## 三、结果交付：Nodus如何呈现答案

核心原则：**不创建窗口。卡片按需生成，用完即焚。**

### 三种结果交付形式

#### 1. 语音回应（轻量答案）

适用于"是/否""在哪里""是什么"类查询。

> "Nodus，这个项目的Node版本是多少"
> "Node 22.5.1，在package.json的engines字段指定。"

不需要UI。一句话就够。

#### 2. 结构化卡片（信息展示）

适用于调用图、diff对比、测试结果、依赖关系等需要**视觉承载**的信息。

**卡片特征**：
- 浮于当前工作区之上，不遮挡正在编辑的代码
- 可拖动、可缩放、可固定（临时固定到侧边）
- 任务相关卡片自动关联——调用图和代码diff可能同时出现，因为它们服务于同一个意图
- 任务结束后，相关卡片一键消散（或5分钟无交互自动消散）

**卡片示例——调用图**：

```
┌──────────────────────────────────────────┐
│  refundOrder 调用链路               ✕     │
│                                           │
│  order.controller.ts:postRefund()         │
│    └─ order.service.ts:processRefund()   │
│       └─ payment.service.ts:refundOrder()│
│          └─ payment-gateway.ts:submit...  │
│                                           │
│  ⚠ 3处amount可能为undefined              │
│     bulk-refund.ts:67   [查看 →]         │
│     webhook-stripe.ts:132 [查看 →]       │
│     admin-api.ts:89     [查看 →]         │
│                                           │
│  [展开全部调用] [导出图]                    │
└──────────────────────────────────────────┘
```

#### 3. 内联变更（代码修改）

AI生成的代码修改**不是卡片，不是聊天消息**——直接以diff形式出现在代码中：

```
原代码（灰底）：
  const order = await db.orders.find(orderId);
  const user = await db.users.find(order.userId);

修改后（白底，左侧有绿色标记）：
  const order = await db.orders.find(orderId);
+ if (!order) throw new NotFoundError(`Order ${orderId} not found`);
  const user = await db.users.find(order.userId);
+ if (!user) throw new NotFoundError(`User ${order.userId} not found`);
```

- 每一个变更块是一个独立的**可接受/可拒绝/可编辑**单元
- 键盘导航在变更块之间跳转（Tab/Shift+Tab）
- 人类可以直接在diff上编辑——这是最有价值的训练信号
- 确认后变更合入源文件，diff消散

## 四、反馈与纠错：当Nodus理解错的时候

### 随时可中断

Nodus在执行过程中可以被随时中断：

> "Nodus，帮我重构这个模块——"
> "等等，不是这个模块，是auth模块。"
> "明白，对auth模块进行重构。"

中断是自然语言的一部分。Nodus不需要专门的"取消"按钮——听到"等等""不对""不是这个"就暂停当前理解和执行。

### 渐进式确认（高风险操作）

对于不可逆操作（删除、部署到生产、修改公共API），Nodus采用**渐进式确认**：

1. Nodus："这个修改会影响3个公共API的签名。要我列出所有受影响的调用方吗？"
2. 用户："列出来。"
3. Nodus展示影响范围卡片
4. 用户（看过之后）："继续。"
5. Nodus执行

对于低风险操作（代码补全、格式化、本地测试运行），Nodus不请求确认，直接执行。

### 纠错即学习

每次用户纠正Nodus的行为，都是一条训练信号：

- "不是这个" → 意图理解偏了，记录修正后的正确理解
- "不对，应该用for...of而不是forEach" → 代码风格偏好，记录
- "不需要加这个null检查" → 过度谨慎，记录

## 五、主动推送：什么值得打断用户

### 授权域模型

用户对每个领域单独授权：

| 领域 | 默认状态 | 说明 |
|------|---------|------|
| 代码库 | 仅通知 | CI失败、依赖安全漏洞——通知 |
| 日程 | 仅通知 | 会议提醒——通知 |
| 邮件 | 仅通知 | 重要邮件到达——通知 |
| 监控/告警 | 可打断 | 生产环境异常——打断 |
| 团队协作 | 仅通知 | PR被review、被@——通知 |

- **仅通知**：屏幕边缘呼吸灯，用户自己决定何时看
- **可打断**：屏幕中央出现紧急卡片，需要用户处理

### 打断优先级判断逻辑

系统在决定是否打断用户前，过三道关：

1. **授权关**：这个领域用户授权可以打断吗？
2. **紧迫关**：这件事现在不处理会有实际后果吗？（CI失败了是紧迫的，但如果是feature分支的CI失败不如主分支紧迫）
3. **状态关**：用户当前处于什么状态？（正在心流中→降低打断意愿；刚切换回来→可以提醒）

三道关全部通过才打断。大部分信息停留在呼吸灯级别。

## 六、开发者交互模式总结

一张图概括Nodus for Developers的完整交互模型：

```
          意图表达层                 执行层                  交付层
    ┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
    │                  │    │                 │    │                 │
    │  语音（主）       │    │  语义理解        │    │  语音回应        │
    │  · 导航          │    │  · 意图解析      │    │  · 简单答案      │
    │  · 查询          │    │  · 歧义反问      │    │  · 进度播报      │
    │  · 编排          │    │                 │    │                 │
    │                  │    │                 │    │                 │
    │  键盘（主）       │───▶│  能力编排        │───▶│  结构化卡片      │
    │  · 编写代码       │    │  · 环境调度      │    │  · 调用图       │
    │  · 编辑diff      │    │  · 代码生成      │    │  · 测试结果      │
    │  · 快捷键        │    │  · 流程执行      │    │  · diff对比     │
    │                  │    │                 │    │                 │
    │  选中（隐式）     │    │                 │    │  内联变更        │
    │  · 代码高亮       │    │  持续学习        │    │  · 代码diff     │
    │  · 光标位置       │    │  · 偏好更新      │    │  · 可编辑/确认   │
    │  · 当前上下文     │    │  · 飞轮数据采集   │    │  · 合入源文件    │
    │                  │    │                 │    │                 │
    └─────────────────┘    └─────────────────┘    └─────────────────┘

    所有界面均为卡片形式，任务结束即消散。无持久窗口。无系统托盘。无桌面图标。
```

## 七、与传统OS的交互对比

| 交互维度 | 传统OS | Nodus |
|---------|--------|-------|
| 主要输入 | 鼠标点击 + 键盘快捷键 | 语音 + 键盘写码 + 选中 |
| 界面形态 | 持久窗口 + 桌面 + 菜单 | 按需生成的卡片，用完即焚 |
| 文件浏览 | 文件夹 → 子文件夹 → 文件 | 语义描述直接定位 |
| 应用启动 | 桌面图标 → 双击 → 等待 | 不存在"启动应用"这个概念 |
| 多任务 | Alt+Tab在窗口间切换 | 任务在后台执行，结果按需浮现 |
| 通知 | 弹窗打断，右下角堆积 | 呼吸灯 + 优先级判断 |
| 错误处理 | 弹窗报错，用户自己查 | 系统主动关联原因，给出上下文 |
