# MVP 功能边界

## MVP 目标

让一个开发者打开Nodus，系统自动准备好项目环境、理解代码库结构，开发者能用自然语言（语音或文字）查询代码库的符号、调用关系和变更历史。开发者不需要打开终端、不需要配环境、不需要grep。

**一句话**：用说话的方式理解代码库。环境这件事不让人看见。

---

## MVP 包含（Must Have）

### 1. 环境全托管（基石一）

**用户看到什么**：
打开一个项目目录（或从git clone一个URL），Nodus自动检测项目语言、安装正确的运行时版本、安装所有依赖、配置好开发环境。用户全程不需要输入任何配置命令。

**具体能力**：
- 自动检测项目语言和框架（MVP覆盖：JavaScript、TypeScript、Python。架构保持语言无关，后续语言以插件形式接入）
- 自动安装匹配版本的运行时（通过版本管理器或容器）
- 自动安装依赖（读取package.json/pyproject.toml/Cargo.toml等）
- 环境状态可视化（一个卡片显示"环境就绪"或"正在安装..."）
- 环境异常时的务实降级（"这个依赖在ARM芯片上需要额外处理，正在用Rosetta模式安装"）

**不做**：
- 不追求覆盖所有语言——MVP覆盖JavaScript、TypeScript、Python三种主流语言
- 不支持复杂微服务架构的多服务环境编排
- 数据库/缓存/消息队列等外部服务的自动配置（v2）

### 2. 代码库语义索引（基石二）

**用户看到什么**：
Nodus在后台持续维护对代码库的语义理解。开发者用自然语言发问，系统返回结构化的答案——不是grep结果，是理解后的信息。

**具体能力**：
- 符号索引：所有函数、类、接口、类型、变量定义的解析和定位
- 引用追踪：每个符号的定义位置和所有引用位置
- 调用图：给定一个函数，展示完整调用链路（向上"被谁调用"，向下"调用了谁"）
- 变更历史：给定一个符号，展示它最近的git变更记录
- 自然语言查询（MVP覆盖的查询类型）：

| 查询类型 | 示例 |
|---------|------|
| 定义定位 | "getUserByEmail在哪里定义的" |
| 引用查找 | "refundOrder被哪些地方调用了" |
| 调用链路 | "从login接口到数据库查询的完整链路是什么" |
| 影响分析 | "如果我改了User模型，哪些文件会受影响" |
| 变更历史 | "auth模块最近一周改了什么" |
| 符号概览 | "payment.service.ts里有哪些导出的函数" |

**不做**：
- 不支持跨仓库的语义索引（只索引当前项目）
- 不支持运行时行为分析（只看静态代码，不看运行时）
- 不支持自然语言生成代码（v2）
- 不支持代码质量分析、安全漏洞检测（v2）

### 3. 意图输入（语音 + 文字）

**用户看到什么**：
- 说"Nodus"唤醒 → 说意图 → 等待结果
- 或按Ctrl+Space → 输入文字意图 → 回车 → 等待结果
- 当前选中的代码/光标所在位置自动作为隐式参数

**具体能力**：
- 语音唤醒 + 语音意图识别
- 文字意图输入（无声模式）
- 上下文感知：光标位置、选中的代码自动成为查询的隐式参数
- 歧义反问：系统不确定时生成选项卡片让用户选

**不做**：
- 不支持手势、眼动等多模态输入（v2）
- 不支持多轮对话式的复杂意图（MVP每次查询相对独立）
- 语音不直接生成代码——只做查询和导航

### 4. 结果交付（结构化卡片）

**用户看到什么**：
查询结果以卡片形式浮现在工作区上，展示结构化的信息（调用图、引用列表、变更历史等）。卡片可拖动、可关闭。任务结束后卡片消散。

**具体能力**：
- 调用图卡片：树状展示调用链路，可展开/折叠，可点击跳转到代码
- 引用列表卡片：分组展示所有引用位置，标注文件路径和行号
- 变更历史卡片：按时间线展示git变更，每次commit的diff摘要
- 代码预览卡片：点击任意符号跳转到代码位置，高亮显示

**不做**：
- 不支持内联diff编辑（那是v2的AI代码生成功能）
- 不支持多卡片关联编排（卡片之间不联动）
- 不持久化卡片状态（关闭Nodus后卡片消失）

### 5. 基础系统外壳

**用户看到什么**：
Nodus启动后是一个极简的工作空间。没有桌面图标，没有任务栏，没有窗口管理器。只有：
- 一个干净的代码工作区（代码查看和基础编辑）
- 一个安静的状态指示器（边缘的呼吸灯表示系统状态）
- 唤起后的意图输入条

**具体能力**：
- 代码文件浏览和查看（语法高亮、基础编辑）
- Nodus唤醒指示灯（屏幕边缘的微弱呼吸光效）
- 基础设置界面（麦克风权限、语音/文字偏好、无声模式切换）

**不做**：
- 不做完整的代码编辑器（不替代VSCode的编辑功能——MVP中用户可以同时在Nodus中查看代码，在VSCode中编辑代码。Nodus不是编辑器替代品，是OS层的信息集成者）
- 不做文件管理器、系统设置面板
- 不做多桌面、虚拟桌面

## MVP 明确不做

| 不做的事 | 原因 |
|---------|------|
| AI代码生成/重构 | v2。需要语义索引和飞轮数据基础先跑通 |
| 跨域调试（日志+代码关联） | v2。需要接入外部系统（日志平台、监控） |
| 自动标注训练飞轮 | v2。需要先有AI代码生成才有"修正→标注"的循环 |
| 多设备同步 | v2。MVP在单机上验证 |
| 多语言支持（JS/TS/Python以外） | v2。先用三种主流语言跑通全链路，架构保持语言无关 |
| 团队协作功能 | v2。MVP是单开发者体验 |
| 主动计算/通知推送 | v2。先做好被动响应（用户问→系统答） |
| 外部服务环境（DB/Redis等） | v2。MVP只管理语言运行时和依赖包 |
| 移动端/平板 | v2。MVP只在桌面端（macOS先） |

## MVP 用户完整流程（端到端）

1. 开发者打开Nodus
2. Nodus检测到当前目录是一个项目，自动识别语言（JS/TS/Python）（或用户说"Nodus，克隆github.com/xxx/yyy并打开"）
3. Nodus自动安装对应运行时（Node.js 或 Python）、安装依赖、配置环境（30秒-2分钟，视项目大小）
4. 环境就绪后，Nodus在后台建立语义索引（首次索引时间视项目大小，增量索引实时）
5. 开发者说："Nodus，refundOrder这个函数被哪些地方调用了"
6. Nodus展示调用卡片：14处调用的结构化列表，高亮3处可能有问题的调用
7. 开发者点击某个调用位置 → 代码视图跳转到对应文件和行
8. 开发者说："Nodus，这个函数的调用链路是什么样的"
9. Nodus展示完整的调用图卡片
10. 开发者说："Nodus，auth模块这个月改了什么"
11. Nodus展示变更历史卡片：7次commit的时间线和摘要
12. 开发者看完，卡片消散。继续工作。

## MVP 成功标准

MVP达到以下标准即可进入下一阶段：

1. **环境就绪**：对一个标准JS/TS或Python项目，从打开到环境就绪不超过2分钟，人类不需要输入任何命令
2. **索引准确**：对10万行以内的JS/TS/Python项目，符号索引覆盖率≥95%，调用图准确率≥95%
3. **查询体验**：从用户说出意图到看到结果卡片，不超过3秒（索引已建立的情况下）
4. **生存测试**：作者自己日常使用Nodus进行开发，持续一周——如果自己都不想用，说明方向错了
